# 前端开发者 → Godot 2D 游戏开发学习总纲

## 这份内容适合谁

适合：

* 前端开发者
* 希望主要靠 AI 写代码，自己负责理解、修改、验收
* 目标是做出一个完整的 2D 独立游戏
* 暂时不考虑 3D

这不是一份“先学完再做”的课程表，而是一份**围绕做游戏来学习 Godot 2D 的完整路线**。

---

# 一、先建立正确认知

## 1. 你的学习方式应该是什么

你现在最适合的方式不是：

* 系统啃完 Godot 文档
* 先把 GDScript 学透
* 看大量零散教程

你最适合的方式是：

> 让 AI 先给出可运行版本 → 你负责看懂结构 → 你负责调试和迭代 → 在做功能的过程中补知识

也就是说，你不是先学引擎，再做游戏。
你是：

> 一边做游戏，一边把做这个游戏需要的 Godot 知识补上。

---

## 2. 你在项目里的角色

在 AI 辅助开发模式里，你不是纯程序员，你更像：

* 设计者
* 组装者
* 验收者
* 调试者
* 节奏控制者

AI 的职责：

* 快速产出代码
* 提供结构方案
* 解释报错
* 重构脚本

你的职责：

* 定义需求
* 判断这段代码是不是你真正想要的
* 确认场景关系对不对
* 验证功能有没有跑通
* 确认项目有没有逐渐失控

---

## 3. 你的第一个目标不是“精通 Godot”

你的第一个目标应该是：

> 能用 Godot 做出一个可以完整跑通的 2D 游戏原型

这个原型要至少具备：

* 开始游戏
* 玩家控制
* 敌人出现
* 攻击与受伤
* 胜负条件
* UI 展示
* 一轮完整流程

一旦你能做出这个，你就已经跨过了最难的阶段。

---

# 二、你要掌握的 Godot 2D 全知识地图

下面是你做一个完整 2D 游戏需要掌握的知识，不要求一次全会，但都要逐步覆盖。

## 模块 1：Godot 基础认知

你必须知道：

* 什么是 Project
* 什么是 Scene
* 什么是 Node
* 什么是 Script
* 什么是 Inspector
* 什么是 Signal
* 什么是 Input Map
* 什么是 TileMap
* 什么是 Resource
* 什么是 Autoload

你的前端类比可以这样理解：

* Scene ≈ 一个页面 / 一个组件树
* Node ≈ 页面里的元素 / 逻辑对象
* Script ≈ 绑定在对象上的逻辑
* Signal ≈ 事件机制
* Autoload ≈ 全局单例
* Resource ≈ 配置数据文件

你不需要先背定义，但你要在做功能时反复用到它们。

---

## 模块 2：GDScript 基础

你不需要学到很深，但以下内容必须能看懂：

* 变量与常量
* 函数
* if / match
* for / while
* Array / Dictionary
* class_name
* extends
* @export
* @onready
* signal
* enum
* preload / load
* Node 路径获取
* 实例化场景

你需要达到的目标不是“手写很快”，而是：

> AI 写出来的 GDScript，你能读懂 70% 以上，并能做简单修改。

---

## 模块 3：2D 场景组织

这是做游戏最重要的基础之一。

你必须学会：

* 一个游戏应该怎么拆场景
* 哪些东西单独做成 scene
* 父子节点怎么组织
* 什么时候复用 scene

典型拆法：

* Main 场景
* Player 场景
* Enemy 场景
* Bullet 场景
* UI 场景
* Level 场景
* Pickup 场景

你必须养成的习惯：

* 每个可复用对象都尽量是独立 scene
* 不要把所有东西都堆在一个场景里
* 一个 scene 只负责一个核心职责

---

## 模块 4：输入系统

你需要掌握：

* Input Map 配置
* 键盘输入
* 鼠标输入
* Action 的概念
* `_process()` 和 `_physics_process()` 的区别

2D 动作游戏里常见输入包括：

* 上下左右移动
* 冲刺
* 攻击
* 交互
* 暂停

你必须明白：

* 移动一般放在 `_physics_process()`
* 纯显示逻辑可放在 `_process()`

---

## 模块 5：2D 运动与坐标

你要理解：

* position / global_position
* Vector2
* 方向向量
* normalize
* 距离计算
* lerp
* rotate / look_at
* velocity

这是你后面做下面这些东西的基础：

* 玩家移动
* 子弹飞行
* 敌人追踪
* 摄像机跟随
* 拾取物飞向玩家

---

## 模块 6：碰撞与物理

这是很多新手第一次会卡住的地方。

你必须掌握：

* Area2D
* CharacterBody2D
* StaticBody2D
* CollisionShape2D
* 层与掩码（collision layers / masks）
* body_entered / area_entered
* hitbox / hurtbox 思路

你需要知道：

* 玩家通常用 CharacterBody2D
* 子弹和伤害检测常常用 Area2D
* 墙体和阻挡通常用 StaticBody2D

你要建立的核心理解：

> 碰撞不只是“碰到”，而是“谁检测谁、谁响应谁、谁该造成伤害”。

---

## 模块 7：战斗系统

做一个游戏，战斗系统通常包括：

* 攻击触发
* 子弹生成
* 命中检测
* 扣血
* 死亡
* 特效
* 攻击冷却

你需要学会的内容：

* 如何实例化 Bullet
* 如何给 Bullet 传伤害和方向
* 如何让 Bullet 命中敌人
* 如何让敌人扣血并死亡
* 如何生成击中特效
* 如何做攻击间隔

你不应该一开始就追求复杂武器系统。
你应该先把下面这个最小闭环做通：

> 玩家发射子弹 → 子弹碰到敌人 → 敌人掉血 → 敌人死亡

---

## 模块 8：敌人系统与 AI

2D 游戏里的敌人 AI，不是大模型意义上的 AI，而是规则逻辑。

你需要学会：

* 追踪玩家
* 巡逻
* 在一定范围内激活
* 攻击玩家
* 死亡后掉落
* 精英敌人变体

建议你的第一批敌人就三种：

* 直线追踪型
* 远程发射型
* 高血量慢速型

这样就够支撑一个原型。

---

## 模块 9：状态系统

当你的游戏稍微复杂一点，就会出现状态。

常见状态：

* Idle
* Move
* Attack
* Hurt
* Dead

你需要学会：

* 用变量和 enum 表达状态
* 避免多个状态互相打架
* 在不同状态限制不同操作

无论是玩家还是敌人，都离不开状态管理。

---

## 模块 10：动画系统

2D 游戏常见动画方式：

* AnimatedSprite2D
* AnimationPlayer
* Tween

你需要掌握：

* 角色走路 / 攻击 / 受伤动画
* UI 动画
* 击中闪烁
* 简单缩放 / 位移反馈

如果你素材不足，可以先用：

* 纯色方块
* 缩放
* 旋转
* 闪白
* 抖动
  来代替复杂动画。

---

## 模块 11：数值与数据

这是很多人忽略但非常关键的一块。

你要学会区分：

* 代码逻辑
* 配置数据

例如：

* 玩家基础血量
* 敌人移速
* 武器伤害
* 升级选项数值

这些不应该全部写死在脚本里。

你需要逐步掌握：

* 用 exported variables 配参数
* 用 Resource / 自定义数据结构配置模块
* 用表驱动思维组织升级内容

---

## 模块 12：UI 系统

一个完整 2D 游戏至少要有这些 UI：

* 主菜单
* 游戏内 HUD
* 血量条
* 经验条
* 暂停菜单
* 失败界面
* 胜利界面

你需要学习：

* Control 节点体系
* 锚点与布局
* Label / Button / TextureRect / ProgressBar
* 信号绑定
* 切换界面显示隐藏

如果你未来要上 Steam，UI 不是可选项，是必须项。

---

## 模块 13：关卡系统

2D 游戏的关卡通常至少涉及：

* 地图
* 出生点
* 敌人生成点
* 边界
* 障碍物
* 波次系统

你需要掌握：

* TileMap 基础
* 关卡场景拆分
* 波次或定时刷怪
* 关卡重开
* 下一关切换

---

## 模块 14：摄像机与反馈

你需要学会：

* Camera2D 跟随
* 屏幕震动
* 击中停顿（hit stop，可选）
* 受伤闪烁
* Overload / 危险状态反馈

很多游戏的“爽感”不来自复杂代码，而来自反馈。

---

## 模块 15：音频系统

你需要掌握：

* BGM 播放
* SFX 播放
* 不同事件对应音效
* 音量控制
* 静音开关

如果素材少，可以先：

* 免费音效站
* AI 辅助生成短音效
* 极简电子音

---

## 模块 16：存档与全局数据

一个完整游戏通常需要：

* 保存设置
* 保存解锁内容
* 保存最高分
* 读取玩家进度

你需要学会：

* JSON / ConfigFile / 文件保存
* Autoload 全局管理
* 场景切换时保留数据

---

## 模块 17：游戏流程与状态管理

完整游戏不只是“能打怪”，还要能完整跑流程。

你需要设计：

* 启动游戏
* 进入菜单
* 开始游戏
* 游戏中
* 暂停
* 结算
* 返回菜单
* 再来一局

也就是所谓的“大状态机”。

---

## 模块 18：调试与排错

你很依赖 AI，所以更要会调试。

你必须掌握：

* 看报错行号
* print 调试
* 变量观察
* 逐步定位是场景问题还是脚本问题
* 判断是逻辑错误还是引用错误

你要形成习惯：

* 一次只改一个问题
* 功能最小化验证
* 不要同时改 5 个脚本

---

## 模块 19：项目结构与可维护性

随着项目变大，你需要知道：

* 文件夹怎么分
* 脚本如何命名
* scene 如何命名
* 全局常量和枚举放哪里
* 公共函数放哪里

推荐基础目录结构：

* `scenes/`
* `scripts/`
* `ui/`
* `audio/`
* `art/`
* `data/`
* `autoload/`

---

## 模块 20：导出与发布基础

虽然你现在不一定立刻发布，但你必须知道：

* 如何导出 Windows 版本
* 如何设置分辨率
* 如何设置图标
* 如何测试 build
* 哪些问题只会在导出后出现

这是“把原型变成产品”的必经之路。

---

# 三、你最适合的学习路径：以做一个 2D 游戏为主线

最适合你的学习路线不是按引擎功能学，而是按做一个游戏的顺序学。

推荐你用下面这个项目作为训练主线：

> 一个 2D 顶视角生存射击游戏

为什么选它：

* 美术压力低
* 2D 知识覆盖广
* 很适合 AI 写代码
* 容易做出可玩的版本
* 能覆盖绝大多数基础系统

这个项目能覆盖：

* 玩家移动
* 敌人追踪
* 子弹系统
* 扣血系统
* 升级系统
* UI 系统
* 波次系统
* 游戏结算

---

# 四、按任务推进的完整学习路线

下面开始进入你最需要的部分：

> 每一个任务要学什么、AI 该写什么、你要看懂什么、验收标准是什么

---

## 阶段 1：搭出最小可玩的原型

### 任务 1：创建项目与主场景

目标：

* 建立工程
* 创建主场景
* 跑通一个空场景

你要学的内容：

* 新建项目
* 创建 scene
* 设置主场景
* 保存 scene
* 运行项目

让 AI 帮你做什么：

* 给你一个基础项目目录建议
* 解释 Godot 里的 scene 和 node 关系

你要重点看懂：

* scene 是怎么保存的
* 根节点是什么
* 为什么所有内容都挂在场景树里

验收标准：

* 能成功运行一个主场景
* 你知道如何添加、保存、运行 scene

---

### 任务 2：做玩家移动

目标：

* 玩家可以用键盘移动

你要学的内容：

* CharacterBody2D
* Input Map
* `_physics_process()`
* 方向输入
* velocity
* `move_and_slide()`

让 AI 帮你做什么：

* 写一个 WASD 移动脚本
* 帮你配置输入动作
* 解释每一行代码含义

你要重点看懂：

* 为什么移动逻辑写在 `_physics_process()`
* velocity 是怎么被计算出来的
* normalized 的作用是什么

验收标准：

* 玩家能上下左右移动
* 斜向移动速度正常
* 没有奇怪抖动

---

### 任务 3：做 Camera2D 跟随

目标：

* 摄像机跟着玩家走

你要学的内容：

* Camera2D
* current / enabled
* 平滑跟随

让 AI 帮你做什么：

* 配置一个基础跟随摄像机
* 可选加入平滑

你要重点看懂：

* 摄像机是节点，不是代码里硬算
* 什么情况下需要平滑

验收标准：

* 角色移动时视角稳定跟随

---

### 任务 4：做敌人生成

目标：

* 地图中持续出现敌人

你要学的内容：

* Timer
* 实例化 scene
* 敌人出生位置
* 父节点管理

让 AI 帮你做什么：

* 写一个敌人生成器
* 定时生成敌人实例

你要重点看懂：

* `preload()` 和 `instantiate()` 的用法
* 生成对象应该加到哪里

验收标准：

* 敌人能每隔几秒出现
* 不会全都生成在屏幕中央

---

### 任务 5：做敌人追踪玩家

目标：

* 敌人会向玩家靠近

你要学的内容：

* 获取玩家引用
* `global_position`
* `direction_to()`
* 运动逻辑

让 AI 帮你做什么：

* 写敌人的追踪脚本

你要重点看懂：

* 敌人为什么能知道玩家位置
* 为什么追踪用方向向量

验收标准：

* 敌人能稳定追过来
* 不会乱抖或绕圈

---

### 任务 6：做子弹与自动攻击

目标：

* 玩家自动发射子弹

你要学的内容：

* Bullet scene
* 发射方向
* 计时攻击
* 攻击冷却

让 AI 帮你做什么：

* 写一个自动发射最近敌人的逻辑
* 写一个基础 Bullet 脚本

你要重点看懂：

* 子弹是独立 scene
* 子弹自己移动，不应该靠玩家持续推动
* 攻击冷却怎么控制

验收标准：

* 玩家会定时自动打敌人
* 子弹方向正确

---

### 任务 7：做伤害与死亡

目标：

* 子弹打中敌人，敌人扣血，血空死亡

你要学的内容：

* Area2D 碰撞
* hitbox / hurtbox 基础思路
* HP 变量
* 受伤函数
* queue_free()

让 AI 帮你做什么：

* 写碰撞脚本
* 写敌人的 take_damage 方法

你要重点看懂：

* 伤害应该由谁发起
* 命中后为什么要判断目标类型
* queue_free 的意义

验收标准：

* 子弹命中后敌人扣血
* 敌人血量归零后消失

---

### 任务 8：做玩家受伤与失败

目标：

* 敌人碰到玩家时，玩家掉血，血空游戏失败

你要学的内容：

* 玩家 HP
* 无敌时间（invincible frames，可后做）
* 游戏失败状态

让 AI 帮你做什么：

* 写玩家受伤逻辑
* 写最简单的 game over 逻辑

你要重点看懂：

* 为什么受伤要防止连续瞬间多次触发
* 游戏失败应该是状态切换，不只是 print 一句

验收标准：

* 玩家能掉血
* 血量归零后游戏结束

---

## 阶段 2：让它变成“一个游戏”

### 任务 9：做 UI HUD

目标：

* 屏幕能显示生命值、击杀数、时间等

你要学的内容：

* Control
* Label
* ProgressBar
* UI 更新

让 AI 帮你做什么：

* 写一个基础 HUD 场景
* 将玩家状态同步到 UI

你要重点看懂：

* UI 和世界节点最好分开管理
* UI 更新不应该写得很混乱

验收标准：

* 游戏中能稳定显示关键数据

---

### 任务 10：做经验与升级系统

目标：

* 敌人死亡掉经验，玩家升级

你要学的内容：

* 掉落物实例化
* 拾取逻辑
* 经验值累计
* 升级阈值

让 AI 帮你做什么：

* 写经验球掉落与吸收逻辑
* 写经验条与升级判定

你要重点看懂：

* 数据变化和 UI 变化怎么联动
* 升级不是加数字而已，而是一个系统入口

验收标准：

* 玩家能捡经验升级
* 升级逻辑稳定触发

---

### 任务 11：做升级选择界面

目标：

* 升级后暂停游戏并弹出 3 选 1

你要学的内容：

* 游戏暂停
* 弹窗 UI
* 信号回调
* 选择后应用升级

让 AI 帮你做什么：

* 写一个简单升级面板
* 随机给出 3 个升级项

你要重点看懂：

* 数据和 UI 如何关联
* 为什么升级选择最好数据驱动

验收标准：

* 升级时能弹出选择框
* 选一个后效果生效

---

### 任务 12：做基础模块系统

目标：

* 至少实现 3~5 个升级模块

建议第一批：

* 增加伤害
* 增加攻速
* 子弹变大
* 穿透
* 分裂

你要学的内容：

* 玩家属性结构
* 模块应用逻辑
* 子弹参数传递

让 AI 帮你做什么：

* 设计一个简单模块数据结构
* 写 apply_upgrade 逻辑

你要重点看懂：

* 升级改变的是“数据”，不是每次都重写逻辑

# 五、你每天该怎么学

## 你的日常工作流

建议你每天只做这 5 步：

### 第一步：定义今天只做一个功能

例如：

* 今天只做玩家移动
* 今天只做敌人生成
* 今天只做升级弹窗

### 第二步：让 AI 生成最小版本

你要对 AI 说清楚：

* Godot 4
* 2D 项目
* 需要哪些节点
* 给完整脚本
* 顺便解释每一段代码用途

### 第三步：自己把代码粘进去跑

不要只看不跑。
一定要亲手：

* 建节点
* 绑脚本
* 配参数
* 点运行

### 第四步：看懂这几个问题

每次功能做完，都问自己：

* 这个脚本挂在哪个节点上？
* 它依赖哪些节点？
* 它什么时候被调用？
* 输入和输出分别是什么？

### 第五步：记录错误与心得

形成一个开发笔记：

* 今天做了什么
* 哪段代码不懂
* 哪个错误是怎么解决的

# 六、你问 AI 的正确方式

你依赖 AI，所以提问方式很关键。

## 错误问法

* 教我 Godot
* 教我做游戏
* 写一个完整游戏

这种问题太大，产出的内容不稳定。

## 正确问法

你要问得具体、可执行。

例如：

* 用 Godot 4 写一个 CharacterBody2D 的 WASD 移动脚本，并逐行解释
* 用 Godot 4 实现一个敌人生成器，每 2 秒生成一个 Enemy 场景实例
* 在 Godot 4 中如何让 Bullet 用 Area2D 检测敌人并调用 take_damage
* 帮我把这个 Godot 脚本重构成更适合新手理解的版本
* 帮我检查这个报错，告诉我是节点路径问题还是脚本逻辑问题

---

# 七、你必须避免的 10 个坑

1. 一开始就做太大的系统
2. 一次性让 AI 写太多脚本
3. 不理解场景树就乱连节点
4. 报错后盲改很多地方
5. 把所有逻辑都写进 Player
6. 不做独立 scene，导致无法复用
7. 逻辑和 UI 完全耦合
8. 先追求复杂特效，不先做闭环
9. 不做目录整理，项目迅速变乱
10. 从来不导出 build，只在编辑器里测
