# Godot 2D 学习计划（AI 协作实战版）

> 适用对象：前端开发者，目标是在做中学，完成一个可跑通的 2D 顶视角生存射击原型。

## 0. 学习规则（先统一）

1. 每次只做一个功能，不并行开新坑。
2. 每次都先做“最小可运行版本（MVP）”，跑通再扩展。
3. 每次完成后做 4 个复盘问题：
   - 脚本挂在哪个节点？
   - 依赖了哪些节点/资源？
   - 生命周期里什么时候执行？
   - 输入是什么，输出是什么？
4. 所有功能必须有“验收标准”，通过才进入下一步。

## 1. 阶段路线总览

### 阶段 A（第 1 周）：最小可玩原型

1. 创建项目与主场景
2. 玩家移动（WASD）
3. Camera2D 跟随
4. 敌人定时生成
5. 敌人追踪玩家
6. 子弹与自动攻击
7. 伤害与死亡（敌人）
8. 玩家受伤与 Game Over

目标结果：可以完整打一轮“移动 → 开火 → 杀敌 → 被击败”。

### 阶段 B（第 2 周）：变成“游戏”

9. HUD（血量、击杀、时间）
10. 经验与升级
11. 升级三选一弹窗
12. 升级模块化（伤害/攻速/穿透等）

目标结果：形成“战斗 + 成长”核心循环。

### 阶段 C（第 3 周）：内容与反馈

13. 三种敌人类型
14. 波次/难度曲线
15. 命中反馈（闪烁/震屏/特效）
16. 完整流程（暂停、胜利、失败、重开）

目标结果：完成“可玩且有反馈”的完整局内体验。

### 阶段 D（第 4 周）：工程化与发布基础

17. 项目结构整理
18. 音频与设置菜单
19. 存档与局外数据
20. 导出 Windows 版本

目标结果：原型具备可交付和可维护能力。

## 2. 按步骤执行清单（详细版，含程序 + 素材 + UI + 音频）

### A. 核心玩法骨架（先跑通）

1. 第一步：项目初始化与目录落盘
   - 产出：`scenes/`、`scripts/`、`ui/`、`art/`、`audio/`、`data/` 基础结构
   - 验收：新文件都能在 Godot 文件系统中识别
2. 第二步：主场景与世界根节点
   - 产出：`Main.tscn`（World、Entities、VFX、UIRoot 分层）
   - 验收：项目能启动到主场景，场景树结构清晰
3. 第三步：玩家移动（WASD）
   - 产出：`Player.tscn` + `player.gd`，配置 Input Map
   - 验收：八方向移动速度一致、无抖动
4. 第四步：Camera2D 跟随
   - 产出：玩家子节点 `Camera2D`（含边界或平滑参数）
   - 验收：视角稳定跟随，不眩晕、不丢目标
5. 第五步：敌人生成器
   - 产出：`EnemySpawner`（Timer + 出生规则）
   - 验收：定时刷怪，出生点不贴脸、不全重叠
6. 第六步：敌人追踪行为
   - 产出：`Enemy.gd`（追踪/接近逻辑）
   - 验收：敌人能持续追击，不绕圈卡抖
7. 第七步：自动攻击与子弹
   - 产出：`Bullet.tscn`、玩家自动索敌发射逻辑
   - 验收：冷却正确，子弹方向和目标一致
8. 第八步：伤害、死亡、清理
   - 产出：`take_damage()`、死亡事件、`queue_free()` 生命周期
   - 验收：子弹命中扣血，死亡后对象正确移除
9. 第九步：玩家受伤与失败态
   - 产出：玩家 HP、受击冷却、Game Over 状态
   - 验收：玩家可死亡且一局能结束

### B. HUD 与视觉替换（你提到的重点补齐）

10. 第十步：临时 HUD（功能优先）
   - 产出：生命、击杀、时间三项 HUD
   - 验收：数值实时同步，不错位、不丢更新
11. 第十一步：UI 布局重构（适配分辨率）
   - 产出：锚点、容器布局、最小分辨率适配
   - 验收：16:9 与 16:10 不出现关键 UI 裁切
12. 第十二步：HUD 视觉资源替换（第一轮）
   - 产出：血条框、图标、数字字体、按钮样式
   - 验收：占位 UI 被替换，风格统一
13. 第十三步：玩家/敌人素材替换（第一轮）
   - 产出：角色贴图或精灵表，替换占位方块
   - 验收：实体可识别，朝向/缩放正常
14. 第十四步：子弹与命中特效素材替换
   - 产出：子弹贴图、命中闪光或爆点特效
   - 验收：战斗可读性明显提升，不卡顿
15. 第十五步：地图与地表素材替换
   - 产出：TileMap 贴图、基础地形风格
   - 验收：地图不再“纯色底”，角色与地表对比清晰

### C. 成长系统与局内循环

16. 第十六步：经验掉落与拾取
   - 产出：经验球 scene、吸附/拾取逻辑
   - 验收：击杀后有掉落，靠近可吸收
17. 第十七步：经验条与升级判定
   - 产出：经验累积、升级阈值、等级显示
   - 验收：经验满后稳定触发升级
18. 第十八步：升级三选一弹窗
   - 产出：暂停游戏 + 3 选 1 UI
   - 验收：选择后恢复游戏并立即生效
19. 第十九步：升级模块化数据结构
   - 产出：升级数据表（Resource/Dictionary）+ `apply_upgrade`
   - 验收：新增升级项不需要大改旧逻辑

### D. 内容扩展与反馈打磨

20. 第二十步：三种敌人变体
   - 产出：追踪型、远程型、坦克型
   - 验收：三者手感和威胁差异明显
21. 第二十一步：波次与难度曲线
   - 产出：随时间调整刷怪频率/上限/种类
   - 验收：5 分钟内难度可感知上升
22. 第二十二步：打击反馈增强
   - 产出：受击闪烁、击中停顿（可选）、屏幕轻震
   - 验收：命中反馈清晰，不影响可操作性
23. 第二十三步：局内流程闭环
   - 产出：暂停、继续、失败、重开、返回菜单
   - 验收：完整一局无死路、无状态错乱

### E. 音频、设置、工程化与发布

24. 第二十四步：音频资源接入（第一轮）
   - 产出：BGM、攻击音效、受击音效、按钮音效
   - 验收：关键事件都有声音反馈
25. 第二十五步：音频总线与设置菜单
   - 产出：主音量/BGM/SFX 滑条 + 静音开关
   - 验收：设置实时生效且可保存
26. 第二十六步：存档与局外数据
   - 产出：设置存档、最高分/解锁数据
   - 验收：重启游戏后数据可恢复
27. 第二十七步：HUD 视觉资源替换（第二轮抛光）
   - 产出：统一字体、色板、间距规范、图标尺寸规范
   - 验收：UI 视觉一致，无明显“拼凑感”
28. 第二十八步：性能与资源体检
   - 产出：节点数量、特效数量、音频并发、纹理尺寸检查
   - 验收：中低配机器可稳定运行（无明显掉帧）
29. 第二十九步：项目结构与命名重构
   - 产出：脚本/场景命名统一，删除废弃资源
   - 验收：新同学可在 10 分钟内看懂工程结构
30. 第三十步：导出配置与发布包测试
   - 产出：Windows Export Preset、图标、分辨率配置
   - 验收：导出包可独立运行，关键功能可用

### F. 资源替换专项（可穿插执行）

31. 第三十一步：资源命名与规格规范
   - 产出：如 `ui_hp_bar_fill.png`、`sfx_hit_01.wav` 的统一命名规则
   - 验收：资源路径可预测，查找成本低
32. 第三十二步：占位资源下线清单
   - 产出：列出所有占位素材并逐项替换或删除
   - 验收：构建产物中无临时占位图/测试音效

### G. 必补关键步骤（保证“能跑通”与“能扩展”）

33. 第三十三步：胜利条件与结算评分
   - 产出：胜利触发条件（存活时长/击杀数/Boss 击败其一或组合）
   - 验收：一局存在“失败”和“胜利”两种明确结束方式
34. 第三十四步：游戏全局状态机（强烈建议）
   - 产出：`MainMenu -> InGame -> Paused -> Result` 的状态切换控制
   - 验收：暂停、升级弹窗、结算、重开不会互相打架
35. 第三十五步：回归测试清单与冒烟测试
   - 产出：10~20 条固定测试项（重开、暂停、升级、音量、存档）
   - 验收：每次改动后 10 分钟内可完成一次基础回归
36. 第三十六步：关卡扩展预留接口
   - 产出：`LevelConfig`（刷怪池、波次参数、地图主题）数据化配置
   - 验收：新增关卡主题主要通过改配置完成，而不是重写核心逻辑
37. 第三十七步：可复现随机种子（进阶）
   - 产出：`run_seed` 与随机系统初始化入口
   - 验收：同一 seed 可复现近似同局流程，便于调试和平衡
38. 第三十八步：学习成果归档
   - 产出：`learning_notes/`（每步问题、修复、截图、代码片段）
   - 验收：可以回溯“我在哪一步学会了什么”

## 3. 每日学习模板（固定 60-120 分钟）

1. 10 分钟：明确今天只做哪一步（不加需求）。
2. 25-60 分钟：实现最小功能并跑通。
3. 15 分钟：修错 + 验收。
4. 10 分钟：复盘记录（问题、收获、下次要改什么）。

## 4. 学习里程碑（检查你是否真的学会）

1. 里程碑 M1（完成第 1-9 步）
   - 你能独立解释 Scene/Node/Script 关系
   - 你能看懂并修改移动、追踪、碰撞脚本
2. 里程碑 M2（完成第 10-19 步）

3. 里程碑 M3（完成第 20-30 步）
   - 你能控制难度曲线并做完整流程管理
   - 你能完成资源替换、音频设置、导出测试
4. 里程碑 M4（完成第 31-38 步）
   - 你能维护资源规范与回归测试清单

## 5. AI 协作约定（我们后续就按这个来）

你每次只要发一句：`做第 N 步`，我将固定提供：

1. 节点结构图（该建哪些节点，挂在哪里）
2. 完整脚本（Godot 4 / GDScript）
3. 逐段注释（解释“为什么这样写”）
4. Inspector 参数清单（你在编辑器里要填什么）
5. 验收步骤（如何确认真的跑通）
6. 常见报错排查（优先给你最可能踩的坑）

固定追加提示词（每一步都建议带上）：

```text
我是学习阶段：你写的所有代码都必须加详细注释。
要求注释覆盖：变量含义、函数职责、关键 API 作用、参数为什么这样设、常见坑与排查点。
```

## 6. 回归测试最小清单（模板）

1. 开始新局后，玩家可移动且摄像机正常跟随。
2. 敌人可正常生成，不会出生在玩家脸上。
3. 子弹可命中并扣血，敌人死亡后被清理。
4. 玩家受伤后不会一帧内掉空血（受击冷却有效）。
5. 升级弹窗出现时游戏暂停，选择后恢复。
6. 重开后计时、击杀、刷怪状态全部正确重置。
7. 音量设置修改后立即生效，重启后仍保留。
8. 胜利与失败都能进入结算界面并返回主菜单。
